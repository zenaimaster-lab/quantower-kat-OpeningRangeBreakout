# KAT Strike v1.0 Architecture & Logic Reference

This document serves as the canonical reference for the v1.0 release of KAT Strike. It records the definitive architecture, core logic, and bug-fix milestones that established full stability across both the Frontend UI and Backend trading engine.

## 1. Frontend UI Architecture (The "3 Pillars" Fix)

The most significant achievement of v1.0 is the complete elimination of MQL5's native UI unresponsiveness, the "1 góc nhỏ" (small corner) click bug, and the double-toggle blinking bug. 

This was achieved by abandoning the flawed `CAppDialog` internal event routing in favor of a mathematically sound **Single Source of Truth** approach based on native Object Clicks.

### The 3 Pillars of Frontend Stability:

1.  **Purging `EVENT_MAP`:**
    *   `EVENT_MAP_BEGIN` and `EVENT_MAP_END` were permanently deleted from `CDashboard`.
    *   This eliminates MQL5's broken internal bounding-box hit testing which caused events to be stolen by invisible or overlapping UI layers.

2.  **Native `CHARTEVENT_OBJECT_CLICK` Interception:**
    *   `OnChartEvent` intercepts `CHARTEVENT_OBJECT_CLICK` **before** it reaches `CAppDialog::ChartEvent()`.
    *   This guarantees that if the user clicks a button, the EA registers it instantly, using the entire surface area of the button object.

3.  **`HandleDirectClick` & Debouncing:**
    *   All button clicks are routed to `CDashboard::HandleDirectClick(const string &objName)`.
    *   A rigid 500ms `GetTickCount()` debounce guard is applied to prevent hardware bouncing or phantom double-click events from `OBJPROP_STATE` resets.
    *   The visual toggle state (`OBJPROP_STATE`) is forced to `false` immediately so the button never gets physically stuck in the "depressed" visual state.

## 2. Backend Engine & Data Synchronization

The bridge between the Frontend UI and Backend execution is managed by the `DashboardParams` struct and the `GetParams()` synchronization method.

### Data Flow Fix (The v0.90 Milestone):
A critical regression was resolved where Backend systems (Countdown, Origami, Risk/Reward) were starved of data because `GetParams()` failed to read the active symbol and timing configurations from the UI fields.

**The Current Data Flow:**
1.  **Initialization:** `OnInit` calls `SetInitialParams()` to establish baseline settings, but the master state lives in the UI fields.
2.  **Synchronization (`GetParams`):**
    *   Extracts `symbol` via `m_lblSym.Text()`.
    *   Extracts timing components (`nyHour`, `nyMinute`, `nySecond`, `utcOffset`, `triggerBeforeSec`).
    *   Extracts trading parameters (SL, TP, Risk %, Timeframe, Trail settings, Origami settings).
3.  **Consumption (`OnTimer` & `OnTick`):**
    *   `OnTimer()` fires every 1 second, fetches the latest `DashboardParams`, and conditionally executes logic.
    *   If `p.symbol` is valid, it proceeds to calculate Real-time Risk/Reward, Total Exposed Lots, and evaluates the `g_timeMgr` for scheduled Auto-Trading.

## 3. Core Logic Modules

### Time Management & Auto-Trading (`CTimeMgr`)
*   Manages the News timing and NYO (New York Open) schedules.
*   Calculates the exact trigger time based on `targetDayOffset`, `nyHour:Minute:Second`, and `triggerBeforeSec`.
*   Continuously feeds the Countdown string to the Dashboard.
*   Executes `g_orderMgr.PlaceOCOOrders()` when the exact trigger second is hit.

### Order Management (`COrderMgr`)
*   Handles execution of Market and Stop orders (OCO).
*   Manages pending order expiration (`CheckExpire`).
*   Provides utilities to Flatten All, Reverse, and Lock positions.

### Origami Engine (`COrigamiMgr`)
*   An advanced scaling-in system based on risk-free thresholds.
*   Initializes only when a new trade is opened (`DEAL_ENTRY_IN`).
*   Calculates 3 add-in steps based on the target growth percent and current balance.
*   Triggers automatically via `OnTimer()` polling when the price reaches defined distances (25%, 50%, 75% to TP).
*   Dynamically trails the Stop Loss of all positions in the bucket to protect the base trade while scaling.
*   **SL Modes:** Can be set to "Don't Move", "Always Original", or "BE + Spread".

### Risk & Reward Calculation
*   **Static Risk:** Computed via `g_riskMgr.CalcRiskRewardInfo` before entering trades. Determines the correct lot size to risk X% of the balance over Y points.
*   **Dynamic Risk:** `OnTimer` loops through open positions matching the Magic Number and Symbol. Computes `totalProfitAtTP` and `totalLossAtSL` based on current active order volumes to provide real-time portfolio exposure.
*   **Breakeven (BE) Line:** Calculates a volume-weighted average entry price, adds/subtracts the spread, and draws a visual yellow line on the chart representing the true Breakeven point for the net position.

## 4. Stability Guarantees

*   **100% Reliable Clicks:** Due to the 3-Pillar UI fix.
*   **Fail-Safe Command Queue:** Operations like "Flatten All" or "Apply Origami" triggered by keyboard shortcuts or buttons are pushed to `m_cmdQueue` and executed sequentially in `OnChartEvent`, preventing MT5 threading collisions.
*   **Graceful Degradation:** If no symbol is detected, the EA safely returns early from core loops, preventing zero-divide or invalid handle errors.
